# AI Node Onboarding Approval URL Contract

Status: Partial
Implementation status: Approval URL generation plus state validation and decision protections implemented; additional hardening/UX follow-ups pending
Last updated: 2026-03-11

## Purpose

This document defines how Core generates and validates onboarding approval URLs for AI Node onboarding sessions.

The URL is safe to display to an operator, including headless node workflows where the node prints a URL and waits for operator action.

## Canonical Format

Status: Partial

Generated URL form:

`<core_base>/onboarding/nodes/approve?sid=<session_id>&state=<state_token>`

Query parameters:
- `sid`:
  - onboarding session identifier
  - references persisted Core-side onboarding session
- `state`:
  - tamper-detection binding token generated by Core
  - matched against persisted session metadata during approval handling

## Server-Side Source Of Truth

Status: Implemented

- Core onboarding session store is authoritative.
- URL parameters are lookup/binding hints; they are not authority state.
- Session decision (`pending|approved|rejected|expired|consumed|cancelled`) must come from server-side persistence only.

## Embedded Vs Stored Data

Status: Implemented

Embedded in URL:
- `sid`
- `state`

Stored only server-side:
- requested node identity metadata
- decision state and actor identity
- expiry timestamp
- approval/rejection timestamps
- linked node ID
- final payload consumption state

Sensitive material must never be embedded in URL:
- trust tokens
- MQTT operational credentials
- internal policy payloads

## Expiry Behavior

Status: Implemented

- Approval page must enforce session `expires_at`.
- Expired sessions become terminal (`expired`) and cannot be approved.
- Expiry handling must return deterministic terminal UX/API responses.

## Decision Semantics

Status: Implemented (baseline)

- Approval URL may be opened multiple times while session is pending.
- Only one terminal decision is allowed:
  - approve or reject once
  - no conflicting subsequent decision
- After terminal decision, approval page should show final state, not allow state mutation.

## Validation Rules

Status: Implemented (baseline)

Core approval handlers must validate:
- session exists (`sid` resolves to persisted session)
- session not expired
- session still pending for decision actions
- `state` token is required and matches stored binding value
- operator is authenticated before decision action
- decision actions pass CSRF checks for cookie-session admin requests

## Current Implementation Notes

Status: Partial

Implemented:
- Core onboarding start endpoint generates `approval_url` with `sid` and random `state`.
- `state` and `protocol_version` are persisted in onboarding session metadata.
- approval page + handler validation enforce required `state`.
- approval/rejection actions enforce single terminal decision semantics.
- decision actions apply CSRF checks for cookie-session admin flows.

Pending:
- explicit invalid/expired URL UX polishing
- additional rate-limit/CSRF tuning based on deployment topology
